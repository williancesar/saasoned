# Atomic Network, the Hard Side, and Seeding — Chen

**Source.** Andrew Chen, *The Cold Start Problem*.

---

## Atomic network

**Definition.** The **smallest possible network that is stable and self-
sustaining** — enough users, of the right types, in one bounded context, that it
delivers value and doesn't collapse when you stop pushing. Growth = creating and
saturating *more atomic networks*, not adding scattered users.

**When it applies.** Any network/marketplace/social/community product,
especially pre-traction or "won't take off."

**When it does NOT.** Products whose value does not depend on other users
(single-player utility) — there is no atomic network; route to channels/
retention.

**Decision cue.** Can you name the bounded smallest unit (this city, this
company, this dorm, this one family) and the minimum participants for it to be
useful unattended? If not, defining it is the first task. **No atomic network ⇒
no scaling discussion** (hub Guardrail 2 — refuse and cite).

**Worked micro-example.** *vai-nascer-v2*: atomic network = one pregnant woman +
her invited family. Not "all pregnant users in Brazil" — that's not atomic,
it's an aggregate of many tiny networks.

**Failure it prevents.** Pouring acquisition into a population that never
reaches local density anywhere — users arrive to an empty room and leave.

---

## The hard side

**Definition.** The minority of users who do the **disproportionate work** the
network depends on (sellers, hosts, drivers, organizers, creators, the paying/
initiating party). They are the hardest to acquire and retain, and the network
is worthless without them.

**When it applies.** Any multi-sided/asymmetric-effort network.

**When it does NOT.** Symmetric peer networks with no heavy side (rare) —
identify if one side still bears more load before concluding this.

**Decision cue.** Ask "who does the work everyone else benefits from?" Acquire
and over-serve *them* first; the easy side follows supply. Acquiring the easy
side first starves the network.

**Worked micro-example.** *vai-nascer-v2*: the hard side is the **pregnant
woman** — family access is tied to her subscription/trial; no her, no value for
anyone. Seed and retain her before optimizing family-member growth.

**Failure it prevents.** A flood of easy-side users with nothing for them to
consume because the hard side was never secured — collapse.

---

## Seeding (getting the first atomic network)

**Definition.** Deliberate, often **unscalable** tactics to manufacture the
first atomic network: hand-recruit the hard side, concentrate launch on one
bounded context, fill the network manually, create initial value/supply
yourself, throttle scope to keep density high.

**When it applies.** Cold Start stage (see the-arc.md); any "how do we get the
network started."

**When it does NOT.** Past Tipping Point with self-sustaining networks —
unscalable seeding is no longer the bottleneck (different stage tactics).

**Decision cue.**

```
No atomic network yet?
├─ Pick ONE bounded context (smallest viable)
├─ Hand-acquire the HARD side there first
├─ Do unscalable things to reach local density
└─ Prove it stays alive unattended  →  only then replicate to the next
Tempted to launch broadly first?  →  STOP; refuse; cite Cold Start Problem
```

**Worked micro-example.** *family-agent*: don't open-launch. Seed by personally
onboarding a handful of families in one community to full working density,
confirm the single-family unit sticks unattended, then replicate family-by-
family — not city-by-city marketing.

**Failure it prevents.** Premature broad launch that spreads users too thin to
ever reach the density that makes the product work.
